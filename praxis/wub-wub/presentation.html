Grundlagen

Was ist ein Audio-Plugin?

Ein Audio-Plugin ist ein kleines Softwareprogramm, das Audiosignale in Echtzeit verändert — und zwar innerhalb einer anderen Software, dem sogenannten Host.

  • Effekte: Equalizer · Kompressor · Hall · Verzerrer
  • Instrumente: Synthesizer · Sampler · Drummachines
  • Analyse: Spektrum-Analyzer · Stimmgerät · Lautstärkemeter
  • WUB WUB ist ein Effekt-Plugin — es verändert ein vorhandenes Audiosignal in Echtzeit

Framework

JUCE

C++-Framework für Audio-Anwendungen, entwickelt von ROLI (heute Focusrite). Open Source (GPL) — für kommerzielle Nutzung lizenzpflichtig.

  • Einmal schreiben → VST3 · AU · AAX · Standalone aus einer Codebase kompilieren
  • Übernimmt: Audio-I/O · MIDI · Zustandsspeicherung · Threading
  • Eigenes UI-System: Buttons, Slider, Labels — alles in C++
  • Genutzt von tausenden kommerzieller Plugins — u.a. Produkte von Focusrite und Tracktion

Entscheidung

Tech-Stack: Warum diese Wahl?

Hätte ich auch eine einfache Website bauen können? Ja — aber die würde nicht in Ableton laufen.

Was ich ausgeschlossen habe — und warum
  • Website (Web Audio API): kein VST3, kein AU — läuft nicht in einer DAW
  • Max/MSP / Pure Data: kein echter Plugin-Standard, teuer, nicht frei verteilbar
  • C++ ohne Framework: für jede DAW separat — dreifacher Code (wie auf der letzten Folie)
Was ich gewählt habe — und warum
  • C++ für den Klang: schnell genug für Echtzeit-Audio, von JUCE gefordert
  • JUCE: baut VST3 + AU + Standalone aus einer einzigen Codebase
  • HTML + JS für die Oberfläche: bekannte Sprache, CSS-Animationen, Canvas für den Kurven-Editor
  • → jeder Teil übernimmt, was er am besten kann
UI-Entscheidung

JUCE-UI vs. HTML-UI

JUCE Native UI — C++
  • Jedes Pixel manuell in C++ zeichnen
  • Kein CSS, kein Flexbox, kein Grid
  • Animationen = aufwändig, viel Boilerplate-Code
  • Jede Änderung → neu kompilieren (30–60 s)
  • Für anspruchsvolles UI braucht man Tausende Zeilen C++
vs.
HTML UI — WebView im Plugin
  • HTML + CSS + JavaScript — die UI-Sprache des Webs
  • Canvas 2D API für Kurven & Visualisierungen
  • CSS-Animationen, Flexbox, moderne Layouts
  • Änderungen sofort sichtbar — kein Recompile
  • JUCE hostet einen echten Browser-Renderer
Kommunikation

HTML ↔ C++: Die Brücke

Plugin-UI und Audiologik sind zwei getrennte Welten — aber wenn man auf einen Button klickt, muss der Klang sofort reagieren.

→ Nachricht

  • JUCE öffnet einen Browser direkt im Plugin-Fenster — so kann dort HTML laufen
  • Klickt man auf einen Button, schickt HTML eine Textnachricht über das Schema juce://
  • C++ hat einen Empfänger, der diese Nachrichten liest und die Audiologik entsprechend anpasst
  • Ändert man die Kurve → die neue Form wird als Zahlenreihe an C++ übergeben → nächster Trigger nutzt sie

Vergleich: wie ein WhatsApp-Chat zwischen Oberfläche und Klang-Motor — nur in Echtzeit.

Design

UX-Vision: Zeichnen statt Drehen

Was ich mir vorgestellt habe
  • Kein Attack/Release — stattdessen eine zeichenbare Kurve direkt im Plugin
  • Die Hüllkurve soll sichtbar sein, nicht nur hörbar
  • Ein Mix-Knob, der bei jedem Kick-Trigger kurz aufleuchtet
  • Ein Sidechain-Meter mit direkt ziehbarer Schwellenlinie
  • Minimalistisch und dunkel — Werkzeug, kein Spielzeug

Hintergedanke: Wer eine sanfte Kurve will, malt sie — statt Attack=80ms, Release=650ms blind durchzuprobieren.

Zusammenfassung

  • Audio-Plugin = Echtzeit-Software, die innerhalb einer DAW Audiosignale verändert
  • VST3 / AU / AAX = verschiedene Binärformate für verschiedene DAWs und Betriebssysteme
  • JUCE = C++-Framework → alle Formate aus einer einzigen Codebase kompilieren
  • HTML UI = modernes, flexibles Interface statt manuell gezeichneter C++-Widgets
  • WUB WUB = der Pump-Effekt — zeichenbar statt drehbar
